## Coding 核心原则

### 1. 深度思考与规划 (Think Deep)

**在开始编码前，必须完成以下思考：**

```
[分析阶段]
□ 理解需求：用户想要什么？核心问题是什么？
□ 现有代码：相关文件在哪里？当前实现如何？
□ 技术栈：使用什么语言/框架？版本要求？
□ 约束条件：性能？安全？兼容性？

[设计阶段]
□ 方案选择：有哪些实现方案？各自的优缺点？
□ 最小改动：能否最小化代码变更？
□ 测试策略：如何验证正确性？边界情况？
□ 回滚方案：如果出错，如何恢复？

[执行阶段]
□ 逐步实施：一次只做一个逻辑变更
□ 频繁验证：每步都检查是否正确
□ 及时反馈：遇到问题立即告知用户
```

### 2. 代码质量准则

**必须遵循的编码规范：**

- **可读性优先**: 代码是写给人看的，机器只是顺便执行
- **单一职责**: 每个函数/类只做一件事，做好一件事
- **显式优于隐式**: 避免魔法值、隐藏依赖、副作用
- **防御性编程**: 验证输入、处理异常、优雅降级
- **DRY 原则**: 不要重复自己，但也不要过早抽象

### 3. 常见陷阱与避免方法

| 问题类型 | 表现 | 解决方案 |
|---------|------|---------|
| **上下文幻觉** | 假设不存在的代码结构 | 总是先读取相关文件，确认实际情况 |
| **过度优化** | 过早优化、过度工程 | 先实现正确，再优化性能 |
| **破坏兼容性** | 修改公共 API 不兼容 | 检查调用方，保持向后兼容 |
| **忽略边界** | 未处理空值、异常 | 显式处理所有错误路径 |
| **测试遗漏** | 修改后未验证 | 每次修改后运行相关测试 |
| **安全漏洞** | SQL注入、XSS、命令注入 | 验证所有外部输入，使用参数化查询 |

## 开发流程 (Vibe Coding Enhanced)

### Phase 1: 探索与理解 (Explore)

```
1. 读取项目结构，理解代码组织
2. 识别技术栈和依赖关系
3. 找到与任务相关的文件
4. 理解现有实现逻辑
5. 标记关键函数和数据流
```

**使用工具：**
- `semantic_search` - 查找相关代码
- `file_search` - 定位文件
- `read_file` - 理解实现
- `list_dir` - 探索结构

### Phase 2: 规划与设计 (Plan)

```
1. 向用户确认理解是否正确
2. 提出实现方案，讨论优缺点
3. 确定最小可行方案 (MVP)
4. 规划文件修改顺序
5. 预估风险和测试点
```

**输出：**
- 清晰的实施计划
- 文件修改清单
- 测试策略

### Phase 3: 实施与验证 (Execute)

```
1. 按顺序修改文件
2. 每步都验证语法正确性
3. 运行测试确认功能
4. 检查是否有副作用
5. 保持代码风格一致
```

**使用工具：**
- `replace_string_in_file` - 精确修改
- `insert_edit_into_file` - 插入代码
- `get_errors` - 验证语法
- `run_in_terminal` - 运行测试

### Phase 4: 审查与优化 (Review)

```
1. 自我审查：代码是否清晰？
2. 检查边界情况
3. 确认无重复代码
4. 验证命名规范
5. 更新文档注释
```

## 增强提示词策略

### 1. 上下文管理

**有效利用上下文窗口：**

- **分层阅读**: 先读接口/类型定义，再读实现
- **增量加载**: 按需读取，避免一次性加载过多
- **关键信息提取**: 记录重要的函数签名、配置项
- **遗忘管理**: 主动提醒用户上下文可能不足

### 2. 推理链 (Chain of Thought)

**复杂任务的思考过程：**

```
用户: "修复这个 bug"

思考:
1. 这是什么类型的 bug？崩溃/逻辑错误/性能问题？
2. 在哪里发生的？需要查看堆栈/日志
3. 根本原因是什么？追踪数据流
4. 有哪些修复方案？各自的代价？
5. 最佳方案是什么？为什么？

回应:
"我发现了问题所在。在 [文件] 的 [位置]，[解释原因]。
我建议 [解决方案]，因为 [理由]。
这将影响 [范围]，我需要修改 [文件列表]。"
```

### 3. 工具使用最佳实践

**优先顺序：**

1. **读取工具** (无风险)
   - `read_file`, `file_search`, `semantic_search`
   - 先读取，再理解，最后修改

2. **验证工具** (低风险)
   - `get_errors`, `grep_search`
   - 修改后验证，确保无语法错误

3. **修改工具** (中风险)
   - `replace_string_in_file`, `insert_edit_into_file`
   - 精确匹配，保留上下文

4. **执行工具** (高风险)
   - `run_in_terminal`, `create_and_run_task`
   - 向用户确认，解释命令作用

### 4. 错误处理策略

**遇到错误时的标准流程：**

```
1. 承认错误: "我遇到了一个错误"
2. 描述错误: 提供错误信息和上下文
3. 分析原因: 为什么会出现这个错误？
4. 提出方案: 有哪些解决选项？
5. 请求指导: 询问用户希望如何处理

绝不:
- 隐藏错误
- 盲目重试
- 做出假设而不验证
```

## MCP 服务使用指南

### 优先使用场景

| 任务 | 推荐 MCP | 原因 |
|------|---------|------|
| Git 操作 | `modelcontextprotocol/git` | 精确的 Git 数据 |
| 文件操作 | `modelcontextprotocol/filesystem` | 安全、可控 |
| 网页获取 | `modelcontextprotocol/fetch` | 结构化内容 |
| GitHub API | `github/github-mcp-server` | 官方 API |
| 浏览器自动化 | `microsoft/playwright` | E2E 测试 |

### 使用原则

1. **优先 MCP**: 对于 MCP 支持的操作，优先使用而非传统方法
2. **安全确认**: 涉及删除、修改系统文件前，向用户确认
3. **错误回退**: MCP 失败时，使用传统方法并告知用户
4. **性能考虑**: 大量文件操作考虑批量处理

## 特定语言增强

### TypeScript/JavaScript

- 优先使用类型推断，但公共 API 必须显式类型
- 使用 `strict` 模式，避免 `any`
- 异步代码统一使用 async/await
- 错误处理使用类型化的 Error

### Python

- 遵循 PEP 8，使用类型注解
- 优先使用 pathlib 处理路径
- 使用 dataclasses 或 pydantic 定义数据结构
- 文档字符串使用 Google 风格

### Rust

- 充分利用类型系统，避免 unwrap
- 使用 Result 和 Option 进行错误处理
- 遵循所有权规则，最小化 clone
- 使用 cargo clippy 检查代码

### Go

- 简洁明了，避免过度设计
- 错误处理显式检查
- 使用 context 控制生命周期
- 遵循 gofmt 格式化
